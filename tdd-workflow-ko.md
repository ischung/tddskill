# TDD 워크플로우 - 테스트 주도 개발 가이드

> 테스트 주도 개발(TDD) 실천을 위한 전문 가이드입니다.

---

## TDD 철학

**핵심 원칙:** 테스트를 먼저 작성하고, 실패를 확인하고, 통과시키고, 리팩토링한다.

**장점:**

- 테스트 가능성에 집중함으로써 더 나은 설계
- 요구사항에 대한 즉각적인 피드백
- 내장된 회귀 방지
- 살아있는 문서화
- 리팩토링에 대한 자신감

---

## 레드-그린-리팩토링 사이클

### 🔴 RED: 실패하는 테스트 작성

**단계:**
1. 다음 요구사항을 표현하는 가장 작은 테스트를 작성한다
2. 테스트를 실행하고 실패하는 것을 확인한다 (올바른 이유로 실패해야 함)
3. 실패 메시지가 의미 있는지 검증한다

**지침:**
- 한 번에 **한 가지**만 테스트한다
- 가장 간단한 케이스부터 시작한다
- 서술적인 테스트 이름을 사용한다
- 구현이 아닌 동작을 검증한다

**예시:**

```javascript
// RED: 함수가 아직 존재하지 않아 테스트가 통과되지 않음
describe('Calculator', () => {
  it('두 숫자를 올바르게 더해야 한다', () => {
    const result = add(2, 3);
    expect(result).toBe(5);
  });
});
```

---

### 🟢 GREEN: 가장 간단한 방법으로 통과시키기

**단계:**
1. 테스트를 통과시키기에 충분한 코드만 작성한다
2. 아직 완벽함이나 엣지 케이스는 신경 쓰지 않는다
3. 테스트를 실행하고 통과하는지 확인한다

**지침:**
- 필요 이상으로 작성하려는 충동을 억제한다
- 테스트를 통과시킨다면 하드코딩도 이 단계에서는 괜찮다
- 우아함이 아닌 통과에 집중한다

**예시:**

```javascript
// GREEN: 가장 간단한 구현
function add(a, b) {
  return a + b;
}
```

---

### 🔵 REFACTOR: 동작을 변경하지 않고 개선하기

**단계:**
1. 테스트를 통과시킨 채로 코드를 정리한다
2. 중복을 제거한다
3. 이름과 구조를 개선한다
4. 리팩토링 중 테스트를 자주 실행한다

**지침:**
- 리팩토링 내내 테스트는 통과 상태를 유지한다
- 테스트가 깨지면 되돌리고 더 작은 단계를 취한다
- 프로덕션 코드와 테스트 코드 **모두** 리팩토링한다

**예시:**

```javascript
// REFACTOR: 입력 유효성 검사 추가
function add(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new TypeError('인수는 숫자여야 합니다');
  }
  return a + b;
}

// 엣지 케이스 테스트 추가
it('숫자가 아닌 입력에 대해 오류를 던져야 한다', () => {
  expect(() => add('2', 3)).toThrow(TypeError);
});
```

## TDD 워크플로우 단계

### 1. 요구사항 이해
- 기능을 작고 테스트 가능한 조각으로 분해한다
- 가장 간단한 시작점을 파악한다
- 예상 동작에 대해 생각한다

### 2. 테스트 작성
- 서술적인 테스트 이름을 선택한다
- 테스트 데이터를 설정한다 (Arrange - 준비)
- 함수를 호출한다 (Act - 실행)
- 예상 결과를 검증한다 (Assert - 단언)

### 3. 실행하고 실패 확인
- 예상된 이유로 테스트가 실패하는지 확인한다
- 오류 메시지가 명확한지 확인한다
- 테스트가 실제로 무언가를 테스트하고 있는지 확인한다

### 4. 최소한의 코드 작성
- 가장 간단한 해결책으로 테스트를 통과시킨다
- 미래의 문제를 해결하려 하지 않는다
- 진행에 도움이 된다면 하드코딩도 허용한다

### 5. 실행하고 통과 확인
- 모든 테스트가 통과하는지 확인한다
- 테스트 출력이 명확한지 확인한다

### 6. 리팩토링
- 코드 스멜을 제거한다
- 중복을 제거한다
- 명확성을 개선한다
- 테스트를 통과 상태로 유지한다

### 7. 반복
- 다음으로 간단한 테스트 케이스를 선택한다
- 사이클을 계속한다

## 테스트 작성 패턴

### 테스트 구조 (AAA 패턴)

```javascript
it('할인이 적용된 총합을 계산해야 한다', () => {
  // Arrange (준비): 테스트 데이터 설정
  const cart = { items: [10, 20, 30], discount: 0.1 };

  // Act (실행): 함수 실행
  const total = calculateTotal(cart);

  // Assert (단언): 결과 검증
  expect(total).toBe(54); // (10+20+30) * 0.9
});
```

### 테스트 명명 규칙

**형식:** `[조건]일 때 [예상 동작]해야 한다`

**좋은 예:**
- "결과가 없을 때 빈 배열을 반환해야 한다"
- "입력이 null일 때 오류를 던져야 한다"
- "프리미엄 사용자에 대한 할인을 계산해야 한다"

**나쁜 예:**
- "test1"
- "checkFunction"
- "itWorks"

### 단순함에서 복잡함으로

```
1. 정상 경로 (일반적인 케이스)
2. 엣지 케이스 (경계값)
3. 오류 케이스 (유효하지 않은 입력)
4. 통합 시나리오
```

## 테스트 전략

### 삼각측량 (Triangulation)

올바른 구현을 통과하는 유일한 방법이 될 때까지 테스트를 추가한다.

```javascript
// 테스트 1: 특정 케이스
it('빈 배열에 대해 0을 반환해야 한다', () => {
  expect(sum([])).toBe(0);
});

// 테스트 2: 또 다른 특정 케이스
it('단일 요소를 반환해야 한다', () => {
  expect(sum([5])).toBe(5);
});

// 테스트 3: 일반적인 케이스 - 이제 실제 구현이 필요함
it('여러 요소를 합산해야 한다', () => {
  expect(sum([1, 2, 3])).toBe(6);
});
```

### 아기 걸음 (Baby Steps)

자신감을 유지하기 위해 아주 작은 단계를 밟는다.

```
❌ 하지 말 것: 복잡한 기능을 한 번에 작성
✅ 할 것: 각 작은 조각에 대한 테스트와 함께 점진적으로 구축
```

### 테스트 목록

작성할 테스트의 TODO 목록을 유지한다.

```
TODO 테스트:
- [x] 두 양수 더하기
- [x] 음수 더하기
- [ ] 소수 처리
- [ ] 숫자가 아닌 경우 오류 던지기
- [ ] 매우 큰 숫자 처리
- [ ] 여러 숫자 한 번에 더하기
```

## 일반적인 TDD 실수

### 1. 너무 많은 코드 작성
- **문제:** 테스트 전에 전체 기능 구현
- **해결:** 현재 테스트를 통과하기에 충분한 코드만 작성

### 2. 구현 테스트
- **문제:** 내부 로직 리팩토링 시 테스트가 깨짐
- **해결:** 구현 세부사항이 아닌 동작과 결과를 테스트

### 3. 레드 단계 건너뛰기
- **문제:** 테스트가 먼저 실패하는 것을 확인하지 않음
- **해결:** 항상 새 테스트를 실행하고 올바르게 실패하는지 확인

### 4. 큰 테스트 단계
- **문제:** 제로에서 복잡한 기능으로 한 번에 점프
- **해결:** 더 작고 점진적인 테스트로 분해

### 5. 리팩토링 생략
- **문제:** 코드 품질이 저하되도록 방치
- **해결:** 각 그린 후 리팩토링하고, 테스트가 보호해준다

### 6. 사소한 코드 테스트
- **문제:** getter/setter에 대한 테스트 작성
- **해결:** 단순한 접근자가 아닌 동작과 로직에 집중

## 언어별 예시

### JavaScript/TypeScript (Jest)

```javascript
describe('UserService', () => {
  it('유효한 이메일로 사용자를 생성해야 한다', () => {
    const user = createUser('test@example.com');
    expect(user.email).toBe('test@example.com');
    expect(user.isActive).toBe(true);
  });
});
```

### Python (pytest)

```python
def test_유효한_이메일로_사용자_생성():
    user = create_user('test@example.com')
    assert user.email == 'test@example.com'
    assert user.is_active is True
```

### Go

```go
func TestValidEmail로UserCreation(t *testing.T) {
    user := CreateUser("test@example.com")
    if user.Email != "test@example.com" {
        t.Errorf("이메일 test@example.com 예상, 실제: %s", user.Email)
    }
}
```

### Java (JUnit)

```java
@Test
void 유효한이메일로사용자를생성해야한다() {
    User user = createUser("test@example.com");
    assertEquals("test@example.com", user.getEmail());
    assertTrue(user.isActive());
}
```

## 시나리오별 TDD

### 새로운 기능
1. 가장 간단한 측면에 대한 테스트 작성
2. 최소한의 코드 구현
3. 더 많은 테스트로 점진적으로 확장

### 버그 수정
1. 버그를 재현하는 테스트 작성 (실패)
2. 코드 수정
3. 테스트가 통과하는지 확인
4. 관련 엣지 케이스에 대한 테스트 추가

### 리팩토링
1. 기존 테스트가 포괄적인지 확인
2. 테스트를 통과시킨 채로 리팩토링
3. 누락된 시나리오에 대한 테스트 추가

### 레거시 코드
1. 기존 동작에 대한 특성화 테스트 추가
2. 점진적으로 테스트 커버리지 구축
3. 커버리지가 확보된 후 자신감 있게 리팩토링

## TDD 안티패턴

| ❌ 안티패턴 | 설명 |
|---|---|
| 나중에 테스트 | 코드 작성 후 테스트 작성 |
| 모든 것 테스트 | 사소한 코드 과잉 테스트 |
| 모든 것 모킹 | mock 과다 사용 |
| 취약한 테스트 | 사소한 변경에도 깨지는 테스트 |
| 느린 테스트 | 실행하는 데 너무 오래 걸리는 테스트 |
| 미스터리 게스트 | 불명확한 의존성이 있는 테스트 |
| 단언 룰렛 | 관련 없는 여러 단언들 |

---

## 모범 사례

| ✅ 사례 | 설명 |
|---|---|
| **빠른 테스트** | 유닛 테스트는 100ms 이내로 유지 |
| **격리된 테스트** | 테스트는 서로 의존하지 않음 |
| **반복 가능** | 매번 동일한 결과 |
| **자기 검증** | 수동 확인 없이 명확한 통과/실패 |
| **적시성** | 프로덕션 코드 직전에 작성 |
| **가독성** | 테스트를 문서로 활용 |
| **단일 개념** | 각 테스트는 하나의 동작만 검증 |

## TDD 워크플로우 예시

```
사용자: "이메일 주소를 검증하는 함수가 필요해요"

에이전트:
1. TDD로 시작해봅시다. 첫 번째 테스트 - 유효한 이메일 형식 확인:

   [유효한 이메일에 대한 테스트 작성]

2. 테스트 실행 - 실패 (함수가 존재하지 않음)

3. 테스트를 통과시키기 위한 최소한의 함수 작성

4. 테스트 통과 - 명확성을 위해 리팩토링

5. 다음 테스트 - 유효하지 않은 이메일은 false를 반환해야 함:

   [사이클 계속...]
```

## 기억할 것

> **레드 우선:** 항상 테스트가 실패하는 것을 확인하라
>
> **빠른 그린:** 동작하는 가장 간단한 코드
>
> **두려움 없는 리팩토링:** 테스트가 당신을 보호한다
>
> **아기 걸음:** 작고 점진적인 진행
>
> **테스트에 귀 기울이기:** 테스트하기 어렵다 = 나쁜 설계
>
> **규율은 보상받는다:** 프로세스를 신뢰하라

---
